## Цель

Cubectl является web-панелью для управления серверными инстансами через backend API

Основной пользовательский сценарий: пользователь открывает web-интерфейс, проходит авторизацию, создает инстанс, управляет жизненным циклом, файлами, логами, метриками и доступами

## Архитектурное решение

Проект сразу проектируется как микросервисная архитектура

Runtime-сервисы:

- `gateway`
- `identity-service`
- `instance-service`
- `file-service`
- `metric-service`

Документационный модуль:

- `docs`

Java/Kotlin-пакеты сервисов должны начинаться с `org.cubectl.*`:

- `org.cubectl.gateway`
- `org.cubectl.identity`
- `org.cubectl.instance`
- `org.cubectl.file`
- `org.cubectl.metric`

## Сервисы

Ответственность сервисов описана в [микросервисах](../architecture/services.md)

Межсервисное REST-взаимодействие выполняется через Spring Cloud OpenFeign

## REST API

Base path: `/api/v1`

Публичные URI проектируются как ресурсы: существительные, множественное число, lowercase, дефисы в path segments, без расширений файлов и без CRUD-глаголов в URI

Snake case используется для JSON-полей, query parameters, enum values и error codes

OpenAPI-файлы находятся в [docs/api/openapi](../api/openapi/index.md)

## Авторизация

OAuth2-интеграции не входят в текущую архитектуру

Авторизация строится вокруг никнейма и пароля без email-верификации

После входа система выдает JWT

## Данные

Для всех сервисов поднимается один контейнер PostgreSQL

Внутри контейнера создаются отдельные базы данных для сервисов

## Docker

`instance-service` управляет контейнерами через ограниченный слой доступа к Docker Engine API

Контейнеры считаются расходниками, данные инстанса хранятся на хосте и монтируются внутрь контейнера

## Файлы

`file-service` отвечает за поиск контента, выбор версий, скачивание файлов, хеши, моды, плагины, дата-паки и интеграции с внешними каталогами

Для Modrinth и CurseForge используется архитектура провайдеров с отдельными доменными пакетами

Generic file upload не входит в MVP

## Метрики и логи

`metric-service` каждые 15 секунд собирает текущую runtime-информацию и пишет снимки в PostgreSQL

В MVP web-интерфейс показывает текущие значения без графиков

Живые runtime-обновления и живые Docker-логи отдаются через SSE

## Web-интерфейс

Web-интерфейс должен покрывать создание и управление инстансами, пользователями, доступами, контентом, логами, runtime-информацией и лимитами ресурсов

Распределение экранов описано в [экранах web-интерфейса](../ui/screens.md)
