# Данные и PostgreSQL

## PostgreSQL

Для постоянных данных используется один контейнер PostgreSQL

Внутри контейнера создаются отдельные базы данных:

- `cubectl_identity`
- `cubectl_instance`
- `cubectl_file`

Каждый сервис подключается только к своей базе данных

Общие таблицы между сервисами не используются

## Владение данными

Детальные схемы таблиц описаны в [схемах баз данных](../database/index.md)

`identity-service` владеет:

- Пользователями
- Ролями
- Permissions
- Глобальными назначениями ролей
- Metadata JWT

`instance-service` владеет:

- Инстансами
- Конфигурацией среды выполнения
- Портами
- Lifecycle events
- Resource limits
- Instance-level access
- Instance content model

`file-service` владеет:

- Metadata контента
- Хешами
- Metadata внешних провайдеров
- Metadata зависимостей
- Записями скачиваний

Redis хранит:

- Runtime snapshot инстанса
- Временное состояние контейнера

Формат ключа Redis:

```text
runtime:instance:{instanceId}
```

TTL runtime-ключей составляет от 60 до 120 секунд

Runtime-состояние можно потерять без потери пользовательских данных

## Резервное копирование PostgreSQL

Нужно предусмотреть регулярный backup всех баз данных внутри контейнера PostgreSQL

Требования:

- Backup по расписанию
- Хранение нескольких последних копий
- Отдельная директория для backup-архивов
- Проверка успешности backup
- Ручной запуск backup администратором позже
- Документированный restore-процесс

Backups миров не входят в обязательный план, так как данные инстансов монтируются как локальные директории на сервере
